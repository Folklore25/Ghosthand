## Summary

Phase 25.5 is a strict correction pass, not another 1.4.0 design round. The current baseline already exposes the right surfaces; the remaining work is to remove current-state dishonesty and clean residual maintainability debt without moving ownership boundaries back toward a flat runtime.

Two correctness defects are concrete and already visible in code. Notification post uses `notify(tag, id, notification)` while cancel uses `cancel(id)` and then drops the buffered entry using a fixed tag, so a tagged post can remain live while Ghosthand reports success and deletes its local evidence buffer ([NotificationDispatcher.kt](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/java/com/folklore25/ghosthand/notification/NotificationDispatcher.kt#L103), [NotificationDispatcher.kt](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/java/com/folklore25/ghosthand/notification/NotificationDispatcher.kt#L172), [NotificationBuffer.kt](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/java/com/folklore25/ghosthand/notification/NotificationBuffer.kt#L58)). Screenshot contract drift is also explicit: the provider comments require MediaProjection foreground-service support, but the manifest and runtime start flags still declare only `dataSync` ([MediaProjectionProvider.kt](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/java/com/folklore25/ghosthand/integration/projection/MediaProjectionProvider.kt#L55), [AndroidManifest.xml](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/AndroidManifest.xml#L12), [AndroidManifest.xml](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/AndroidManifest.xml#L89), [GhosthandForegroundService.kt](/Users/zohar/Code/Project/ongoing/Ghosthand/app/src/main/java/com/folklore25/ghosthand/service/runtime/GhosthandForegroundService.kt#L87)).

The rest of the phase should remain mechanical and contained: clear `89` wildcard-import files in main source, choose one or two extraction targets only, and continue obvious test moves from the `16` remaining root-flat tests. The safest plan is to front-load correctness, then run hygiene, then do minimal contained extractions, and finish by moving only tests whose ownership is already unambiguous.

**Primary recommendation:** execute Phase 25.5 as five narrow plans in this order: notification truth, screenshot contract alignment, wildcard-import cleanup, contained extraction pass, test-package convergence plus regression hardening.

## Common Pitfalls

### Pitfall 1: Buffer Truth Is Fixed But Runtime Cancel Still Uses The Wrong Identity
**What goes wrong:** the buffer removal becomes more selective, but `NotificationManager.cancel(id)` still misses tagged notifications.
**How to avoid:** make cancel use the same identity pair used for post and only remove the buffered entry after that exact identity is targeted.

### Pitfall 2: Manifest And Runtime Flags Are Updated But Capability Text Still Says Something Different
**What goes wrong:** runtime behavior becomes more truthful while `/state`, `/capabilities`, or `/commands` still advertise old semantics.
**How to avoid:** treat screenshot alignment as one workstream spanning manifest, service flag, capability payloads, availability projection, and catalog wording.

### Pitfall 3: Wildcard Cleanup Turns Into Phase-Wide Code Motion
**What goes wrong:** mechanical import edits get mixed with unrelated code cleanups, creating noisy diffs and hiding regressions.
**How to avoid:** keep the import pass behavior-neutral and separate from extraction work.

### Pitfall 4: Contained Extractions Drift Into Accessibility-Service Redesign
**What goes wrong:** the largest files invite a deeper refactor of `GhostCoreAccessibilityService` or `GhostAccessibilityService`.
**How to avoid:** do not touch those unless a defect fix requires it. Prefer smaller wins in `RuntimeStateStore` or `GhosthandCommandCatalogRoutes`.

### Pitfall 5: Test Convergence Repackages Ambiguous UI Or Misc Tests Prematurely
**What goes wrong:** package moves become subjective and consume the phase.
**How to avoid:** move only the tests whose owner package is already obvious from current source layout and assertions.
