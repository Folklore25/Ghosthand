# Phase 24.1 Architecture Fix Note — Maintainability Convergence Review Fix Pass

## Purpose

Bind the corrective pass to the exact failures that rejected the first 1.3.1 maintainability review.
This note is intentionally stricter than Phase 24's architecture note: it defines what must become smaller, where ownership must move, how packages must express that ownership, how tests must move with the code, and how the non-root execution or observation seam must become materially stronger before any broader refactor claims count.

## The Rejected Failures This Note Corrects

1. `LocalApiServer` still operates as the practical runtime control center.
2. `StateCoordinator` still operates as a mega-coordinator.
3. State, effect, summary, hint, fallback, and preview layers still overlap too much.
4. Runtime packages remain too flat to enforce ownership.
5. Test ownership remains too flat to guide future maintenance.
6. Route or contract logic still knows too much about present execution and observation details.

Every corrective change in Phase 24.1 must map back to at least one of these failures.

## LocalApiServer Must Become A Thin Shell

`LocalApiServer` keeps only:
- server lifecycle and socket startup or shutdown
- HTTP request parsing at the envelope level
- shared request validation and route lookup orchestration
- dispatch into domain-owned route handlers
- shared response writing and top-level error translation

`LocalApiServer` must stop owning:
- domain-specific route behavior
- per-route disclosure or fallback decisions
- command-catalog assembly details
- route-specific payload shaping
- route-specific execution or observation branching

Target filesystem direction:
- `app/src/main/java/com/folklore25/ghosthand/server/LocalApiServer.kt`
- `app/src/main/java/com/folklore25/ghosthand/server/HttpRequestParser.kt`
- `app/src/main/java/com/folklore25/ghosthand/server/RouteDispatcher.kt`
- `app/src/main/java/com/folklore25/ghosthand/routes/action/*`
- `app/src/main/java/com/folklore25/ghosthand/routes/read/*`
- `app/src/main/java/com/folklore25/ghosthand/routes/input/*`
- `app/src/main/java/com/folklore25/ghosthand/routes/system/*`
- `app/src/main/java/com/folklore25/ghosthand/routes/wait/*`

Correction rule:
adding or changing a route should primarily touch a route-domain package, not `LocalApiServer`.

## StateCoordinator Must Become A Thin Composition Shell

`StateCoordinator` keeps only:
- composition of observation, screen-shaping, preview, state-summary, and capability collaborators
- orchestration of cross-domain calls when a response needs multiple derived layers
- top-level normalization where multiple collaborators must agree on one canonical concept

`StateCoordinator` must stop owning:
- direct screen-summary shaping logic
- direct full-screen payload shaping logic
- direct preview assembly
- direct capability and permission summary assembly
- direct post-action state-summary derivation when it can live in a state-domain helper
- mixed fallback or hint decisions spread across unrelated branches

Target filesystem direction:
- `app/src/main/java/com/folklore25/ghosthand/state/StateCoordinator.kt`
- `app/src/main/java/com/folklore25/ghosthand/state/summary/*`
- `app/src/main/java/com/folklore25/ghosthand/screen/read/*`
- `app/src/main/java/com/folklore25/ghosthand/screen/summary/*`
- `app/src/main/java/com/folklore25/ghosthand/preview/*`
- `app/src/main/java/com/folklore25/ghosthand/capability/*`
- `app/src/main/java/com/folklore25/ghosthand/interaction/effects/*`

Correction rule:
changing one behavior area such as preview, screen summary, or state summary should not require reading a giant multi-domain coordinator first.

## Layer Convergence Rules

These layer homes must become explicit and non-overlapping.

### Action effect

Owns only immediate action outcome and before/after observation deltas.

Canonical home:
- `interaction/effects`

Must not also act as:
- post-action state summary
- screen summary
- fallback or retry explanation

### Post-action state summary

Owns only "what state am I in now?" after an action completes.

Canonical home:
- `state/summary`

Must not also own:
- immediate effect delta fields
- full-screen element content
- preview transport

### Screen summary

Owns only lightweight current-surface orientation.

Canonical home:
- `screen/summary`

Must not also own:
- full-screen tree payloads
- post-action state semantics
- route fallback policy

### Full screen

Owns only the detailed structured current surface.

Canonical home:
- `screen/read`

Must not also own:
- compact summary heuristics
- action outcome claims

### Hint, disclosure, retry, and fallback

Own only bounded explanation of observation limits or justified retry suggestions.

Canonical home:
- colocated with the domain that truthfully knows why the hint exists, usually `screen/read`, `screen/summary`, or `wait`

Constraint:
the same hint field must not be independently reinterpreted by action-effect, state-summary, and screen-summary codepaths.

### Preview metadata and access

Owns only preview availability, tokens, and preview lookup helpers.

Canonical home:
- `preview`

Must not be shaped ad hoc from screen-summary or state-summary code.

## Package Convergence Rules

Phase 24.1 must move from a mostly flat package to domain-owned packages with real responsibility boundaries.

Minimum package convergence target:
- `server`
- `routes`
- `payload`
- `state`
- `screen`
- `interaction`
- `preview`
- `capability`
- `catalog`
- `wait`

Packaging rules:
- package movement must reflect ownership, not aesthetic file shuffling
- one dominant runtime concept gets one canonical package home
- helpers that remain imported by many domains should live under the owning domain, not a new generic utility bucket
- files that still centralize multiple domains after moving packages count as unresolved convergence

## Stronger Execution And Observation Seam

Phase 24 created an initial seam. Phase 24.1 must make it operationally stronger.

Route or contract code may know:
- request models
- response models
- route descriptors
- which execution or observation collaborator to ask for a capability

Route or contract code must not know:
- accessibility observation traversal details
- action execution mechanics beyond collaborator contracts
- preview or screenshot storage details
- how root might work later

Execution-side collaborators should concentrate under current non-root domains such as:
- `interaction/execution`
- `interaction/effects`

Observation-side collaborators should concentrate under domains such as:
- `screen/read`
- `screen/summary`
- `state/summary`
- `preview`
- `capability`

Present-value seam rule:
the seam is only justified if it removes today's coupling between routes, execution mechanics, and observation shaping. No speculative root interface should be introduced unless it already reduces current non-root entanglement.

## Test Ownership Fix

Tests must move with runtime ownership so future changes have an obvious home.

Target test structure direction:
- `app/src/test/java/com/folklore25/ghosthand/server/*`
- `app/src/test/java/com/folklore25/ghosthand/routes/action/*`
- `app/src/test/java/com/folklore25/ghosthand/routes/read/*`
- `app/src/test/java/com/folklore25/ghosthand/state/summary/*`
- `app/src/test/java/com/folklore25/ghosthand/screen/summary/*`
- `app/src/test/java/com/folklore25/ghosthand/payload/*`
- `app/src/test/java/com/folklore25/ghosthand/catalog/*`
- `app/src/test/java/com/folklore25/ghosthand/wait/*`

Test rules:
- request parsing tests stay with `server`
- disclosure and response-shape tests stay with the route or read domain that owns those fields
- state-summary tests move under `state/summary`
- screen-summary and full-screen tests split by their actual runtime owners
- package-flat "misc runtime tests" should disappear by the end of the corrective pass

## Corrective Execution Map

1. Land this note first and use it as the acceptance gate for all later plan work.
2. Thin `LocalApiServer` until it reads like server orchestration plus dispatch only.
3. Thin `StateCoordinator` until it reads like composition only.
4. Move state, effect, summary, hint, and preview logic into their canonical homes and delete overlapping variants.
5. Move runtime files into domain packages only where real ownership now exists.
6. Move tests into matching domain packages as runtime ownership stabilizes.
7. Tighten route or contract versus execution versus observation contracts and verify compile plus focused suites after each major wave.

## Acceptance Bar For This Corrective Pass

Phase 24.1 only counts as complete if the codebase makes these statements true:
- `LocalApiServer` is no longer the place maintainers primarily edit for route behavior.
- `StateCoordinator` is no longer the place maintainers primarily edit for screen, state, preview, and capability shaping.
- the current layer names map to single runtime owners instead of overlapping variants
- package paths visibly express runtime ownership
- test paths mirror runtime ownership closely enough to guide future additions
- route or contract code can evolve separately from execution and observation details without speculative root scaffolding
