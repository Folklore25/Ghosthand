# Phase 24.2 Architecture Fix Note — Maintainability Convergence Fix 02

## Purpose

Bind the second corrective pass to the exact three failures that still block strict maintainability review.
This note is the acceptance gate for Phase 24.2. It is not a general refactor brief.
Each later plan in 24.2 must remove one or more of these failures without adding product scope, root work, or new speculative abstractions.

## The Three Remaining Review Failures

1. `StateCoordinator` is still too large and too central to count as converged ownership.
2. New large domain files replaced the old monoliths instead of removing monolithic change points.
3. Canonical ownership of shared state-legibility concepts is still missing, so different paths can silently drift.

All corrective work in 24.2 must map back to at least one of these failures.

## Traceability Map

| Review failure | What must become true | Primary plans |
|---|---|---|
| `StateCoordinator` still central | `StateCoordinator` becomes a narrow orchestration shell with domain-owned collaborators for health, read, find, preview, capability, and execution-adjacent composition | 24.2-02, 24.2-05 |
| New domain monoliths replaced old ones | `ActionRouteHandlers`, `ReadRouteHandlers`, `GhosthandPayloadSupport`, and the remaining heavy coordinator sections are split by stable behavior family, not by arbitrary helper extraction | 24.2-03, 24.2-05 |
| Canonical ownership is missing | `renderMode`, `surfaceReadability`, `visualAvailable`, `previewAvailable`, `focusedEditablePresent`, and reused fallback semantics flow from one derivation path each | 24.2-04, 24.2-05 |

## Failure 1: Thin `StateCoordinator` Into A Real Shell

### What `StateCoordinator` may keep

- orchestration across already-owned collaborators when one response needs multiple domains
- dependency wiring for state, screen, preview, capability, and execution-facing composition
- top-level normalization when multiple collaborators must agree on one canonical field

### What `StateCoordinator` must stop owning

- direct health, info, device, foreground, or post-action payload assembly
- direct screen-read or find-read derivation logic
- direct preview availability or preview token shaping
- direct capability and permission summary shaping
- direct fallback or hint heuristics that belong to a read or state domain

### Required extraction direction

- `state/health/` owns health, info, device, foreground, and state payload composition
- `screen/read/` owns detailed current-surface read composition and OCR or hybrid read orchestration
- `screen/find/` owns find-result read-context derivation
- `preview/` owns preview availability, lookup metadata, and preview access helpers
- `capability/` owns capability and permission truth composition
- `interaction/execution/` remains the present execution seam and may expose only the execution-facing contracts actually needed by routes or state composition

### Guardrail

Do not satisfy this failure by leaving the logic in `StateCoordinator` and hiding it behind new nested helpers.
Ownership must move to domain files that are the first place a future maintainer would edit.

## Failure 2: Reduce Replacement Monoliths By Stable Family

### Priority files called out by review

- `routes/action/ActionRouteHandlers.kt`
- `routes/read/ReadRouteHandlers.kt`
- `payload/GhosthandPayloadSupport.kt`
- remaining oversized sections of `state/StateCoordinator.kt`

### Required split rules

- action routes split by stable action family:
  - click or tap
  - navigation
  - motion or scroll
  - gesture or input if that grouping is cleaner
- read routes split by stable read family:
  - tree or screen
  - screenshot or preview
  - info, state, device, or foreground
- payload support split by payload family:
  - interaction payloads
  - screen payloads
  - disclosure or fallback payloads
  - input payloads

### Guardrails

- no "misc", "common2", or catch-all file names that simply recreate the same problem
- no arbitrary file explosion that spreads one behavior across tiny fragments
- each new file must own one stable behavior family with an obvious test home

The acceptance bar is not "more files exist". The acceptance bar is "a future change to one behavior family lands in one clearly owned place".

## Failure 3: Establish Canonical Ownership Of Shared State-Legibility Concepts

Shared concepts must have one derivation path each. Other domains may project them, but must not silently recompute them with different thresholds or heuristics.

### Canonical ownership rules

| Concept | Canonical owner | Allowed projections | Must stop happening |
|---|---|---|---|
| `renderMode` | `screen/read` read-context derivation | projected into compact post-action state or route responses | independent recomputation in post-action state or route helpers |
| `surfaceReadability` | `screen/read` read-context derivation | projected into summaries and disclosures | separate readability thresholds in state or action payload code |
| `visualAvailable` | `screen/read` plus `preview/` availability truth | projected into summaries and read payloads | ad hoc booleans derived from unrelated preview or screenshot branches |
| `previewAvailable` | `preview/` | projected into screen and state summaries | inferred separately from screenshot success or render-mode guesses |
| `focusedEditablePresent` | state or read-context owner closest to current focused-node truth | projected into post-action state or input-related payloads | parallel derivation in payload helpers and coordinator branches |
| fallback or hint semantics reused across layers | owning domain that knows the limit or retry reason | reused verbatim by other layers | each layer restating or mutating the same hint independently |

### Explicit correction target

`PostActionStateComposer` must stop independently re-deriving render or readability semantics when `ScreenReadPayload` already owns that truth.
Post-action state should depend on a shared read-context projection, not its own threshold logic.

## Preserved Non-Root Seam

The current non-root seam stays and must get stricter during the fix.

### Required boundaries

- route or contract layer owns request or response contracts and route-family dispatch decisions
- execution layer owns present execution mechanics behind the existing non-root execution contracts
- observation and read layer owns screen, preview, capability, and state-legibility truth

### Rules

- route code must not learn execution mechanics beyond the collaborator contract it uses
- state, screen, preview, and payload code must not know route dispatch structure
- no root code, no libsu integration, no shell wrappers, and no future-root placeholder abstractions

This pass strengthens a present-value seam only. If a change does not reduce current non-root coupling, it does not belong in 24.2.

## Corrective Execution Order

1. land this note as the binding acceptance gate
2. thin `StateCoordinator` until its remaining code is orchestration-only
3. split replacement monoliths by stable behavior family
4. replace duplicated state-legibility derivations with canonical ownership and projection
5. verify that package and test ownership still line up with the runtime ownership introduced in 24.1
6. finish with focused verification plus compile and unit-test gates

## Acceptance Bar

Phase 24.2 only succeeds if all of the following are true:

- `StateCoordinator` is materially smaller and no longer the practical control surface for unrelated domains
- `ActionRouteHandlers`, `ReadRouteHandlers`, and `GhosthandPayloadSupport` stop acting like replacement monoliths
- the shared state-legibility concepts above have one obvious source of truth each
- the route versus execution versus observation seam is stronger than before, still non-root, and still free of speculative root work
- package and test ownership stay sharper after the splits instead of flattening again
