# Phase 24.3 Architecture Fix Note — Final StateCoordinator Thinning

## Purpose

Resolve the last strict maintainability-convergence blocker: `StateCoordinator` still acts as too-central coordination surface even after the earlier corrective passes.

This note is binding for the rest of Phase 24.3. Plans `24.3-02` through `24.3-04` must follow this split exactly instead of reopening already-passing areas.

## What This Pass Owns

Only this:
- materially thin `StateCoordinator`
- move remaining concrete responsibilities into clearer collaborator-owned modules
- preserve the already-passing LocalApiServer/package/test/canonical-ownership gains

It does not own:
- new features
- root/libsu work
- LocalApiServer rework
- broader package churn outside what StateCoordinator thinning requires
- new "replacement monolith" helpers that just move the coordinator problem elsewhere

## Binding Execution Split

### Plan 24.3-02

Own only the remaining state/read/preview thinning:
- state/health/info/device/foreground composition
- screen/tree/find/focused-node read composition
- screenshot/preview read ownership

### Plan 24.3-03

Own only the remaining execution and utility thinning:
- interaction execution entrypoints that still read as coordinator-owned
- clipboard/notification/wait/gesture utility accessors
- any leftover thin pass-throughs that still make `StateCoordinator` the default landing zone

### Plan 24.3-04

Own only preservation and verification:
- minimal test updates required by the ownership moves
- final compile and unit-test gates
- confirmation that already-passing areas stayed intact

No later Phase 24.3 plan should reopen LocalApiServer/package/test/canonical-ownership work except for the smallest direct preservation edits required by the coordinator move.

## Required Thinning Direction

`StateCoordinator` should end this pass as:
- a wiring/composition shell
- plus a small number of truly cross-domain orchestration methods

It should stop being the default landing zone for:
- state/health/info/device/foreground payload composition
- screen/tree/find/focused-node read composition
- interaction execution entrypoints
- screenshot/preview access
- clipboard/notification/wait/gesture utility delegations

## Concrete Ownership Moves

### State/health/info/device/foreground

Move end-to-end payload and status composition behind state-owned collaborators.
`StateCoordinator` may invoke those collaborators, but it should not remain the place where these payloads are assembled or derived.

### Screen/tree/find/focused-node

Move read-path ownership behind screen/read and screen/find collaborators.
`StateCoordinator` should stop being the main home for tree lookup, focused-node access, or screen-read composition.

### Interaction execution

Use the existing non-root execution collaborators more directly.
The coordinator should keep only the smallest practical cross-domain orchestration surface instead of route-adjacent execution detail.

### Screenshot/preview

Push screenshot and preview read behavior into preview/read-owned collaborators.
Screenshot fallback selection should stay in the explicit execution seam established earlier rather than drifting back into the coordinator.

### Clipboard/notification/wait/gesture utilities

Reduce direct method-heavy ownership in the coordinator wherever possible.
These should become collaborator-owned behaviors, with coordinator pass-throughs allowed only when a call is genuinely cross-domain and still reads as shell-level wiring.

## Non-Negotiable Shape After The Pass

`StateCoordinator` should read as:
- constructor wiring over explicit collaborators
- a small set of truly cross-domain orchestration methods
- thin delegation where keeping a public facade still has present value

It should not read as:
- the default home for payload shaping
- the default home for screen or state reads
- the default home for utility accessors
- a new indirection layer hiding behavior that still effectively lives in the coordinator

## Acceptance Bar

Success requires:
- materially lower coordinator line count
- materially fewer public methods
- visibly stronger collaborator ownership from imports and delegation
- preserved passing areas outside the coordinator blocker
- a follow-up diff that clearly matches the `24.3-02` / `24.3-03` split above instead of mixing both waves together

## Risk Controls

- do not broaden scope
- do not undo package/test gains
- do not rework LocalApiServer unless directly required by the coordinator move
- do not add speculative abstractions with no present value
- keep compile and focused behavior checks after the thinning
