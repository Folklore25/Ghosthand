# Phase 25.4: Clean Finish And Capability Presentation Polish For 1.4.0 - Context

**Gathered:** 2026-04-04
**Status:** Ready for planning
**Source:** Direct bounded closeout-polish brief after accepted 25.3 architecture convergence

<domain>
## Phase Boundary

This pass owns exactly four workstreams:
- code cleanliness and refactor-residue cleanup
- capability-plane presentation polish
- `/commands`, `/state`, and `/capabilities` coherence polish
- final 1.4.0 closeout cleanup

It does not own:
- new product features
- new routes except what is strictly required to polish already-accepted behavior
- capability-plane redesign
- observation-plane work
- intent-layer work
- root/libsu or 2.0 work
- 1.4.1 optimization work
- front-end redesign
- app-specific heuristics
</domain>

<decisions>
## Locked Decisions

- Treat the accepted 25.3 architecture result as binding. Do not reopen the accepted package ownership or capability-plane structure.
- This is a polish and closeout pass, not a feature pass and not a new architecture pass.
- Cleanup must preserve behavior and remove only real roughness, residue, or truth-conflicting leftovers.
- Capability-plane work in this pass is presentation polish only: clearer field naming, ordering, grouping, and expression without changing the accepted model direction.
- `/commands` remains the route-contract surface, `/state` remains runtime-oriented, and `/capabilities` remains the capability-plane surface; the task is to polish the coherence between them, not remodel them.
- `docs/API.md` may be aligned only if needed and only in a bounded way. Do not turn the pass into a docs project.
- No de-scoped 1.x capabilities may be reintroduced during cleanup.
</decisions>
