# Phase 25.3: Architecture Convergence Correction Pass For 1.4.0 - Context

**Gathered:** 2026-04-04
**Status:** Ready for planning
**Source:** Direct strict-correction brief after rejected 25.1 architecture-convergence acceptance

<domain>
## Phase Boundary

This correction pass owns exactly two workstreams:
- root-package / directory convergence
- Capability Plane v2 implementation and alignment

It does not own:
- new product features
- root/libsu or 2.0 execution work
- observation-plane redesign
- intent-layer work
- unrelated endpoint growth
- 1.4.1 optimization work
</domain>

<decisions>
## Locked Decisions

- Treat the failed strict acceptance result as binding. Do not defend the prior 25.1 work as sufficient.
- The root package must stop being the default ownership sink; only true top-level entrypoints should remain there after this pass.
- Root-package convergence must move domain-owned UI, service, provider, runtime, execution, screen, notification, and integration classes into real owner packages rather than stopping after easy low-coupling moves.
- Capability Plane v2 must be implemented as three aligned layers: `CapabilityDefinition`, `CapabilityAvailability`, and `CapabilityPresentation`.
- `CapabilityAvailability` must be derived from real runtime state, not copied by hand from route metadata.
- `GET /capabilities` is mandatory in this pass and must be capability-centric rather than a route-centric alias of `/commands`.
- `/commands` remains route-centric, `/state` remains runtime-centric, and `/capabilities` becomes the primary machine-readable capability-plane surface; all three must align from one canonical capability source.
- Stale or truth-conflicting launch-adjacent and other de-scoped 1.x capability remnants must not be re-legitimized during cleanup.
- The pass must stay bounded to the two requested workstreams and must not reopen root/libsu, observation-plane, intent-layer, or unrelated feature work.
</decisions>

<specifics>
## Specific Ideas

- The current root package still contains many classes that clearly belong in `ui/*`, `service/*`, `interaction/*`, `state/*`, `screen/*`, `notification/*`, and `integration/*`.
- The correction pass should explicitly audit remaining root-package files and classify each one as move, keep as true entrypoint, or remove as stale.
- Capability Plane v2 must expose at least:
  - definition fields: `capabilityId`, `domain`, `kind`, `implemented`, `directness`, `truthType`, `preconditions`, `failureModes`, `routes`
  - availability fields: `capabilityId`, `availableNow`, `degraded`, `blockers`, `requiredServices`, `requiredPermissions`, `currentBackend` or `currentMode`
  - aligned presentation for `/capabilities`, `/commands`, and `/state`
- `/capabilities` must expose a compact capability-centric response with definitions, current availability, blockers, preconditions, failure modes, directness, truth type, and route mapping.
- Focused tests must prove root-package convergence expectations, `/capabilities` behavior, capability modeling, cross-surface alignment, and no accidental reintroduction of de-scoped 1.x capabilities.
</specifics>
