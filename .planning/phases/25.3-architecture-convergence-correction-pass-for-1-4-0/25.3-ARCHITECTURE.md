Correct the failed strict 1.4.0 architecture-convergence review by materially executing two things in code:

- Remaining root files must be explicitly classified as `keep`, `move`, or `remove`.
- No de-scoped 1.x public capability line may be re-legitimized during cleanup.

## Capability Plane V2 Canonical Model

Three layers are mandatory and canonical:

### 1. `CapabilityDefinition`

Static capability truth. Each definition must expose:
- `capabilityId`
- `domain`
- `kind`
- `implemented`
- `directness`
- `truthType`
- `preconditions`
- `failureModes`
- `routes`

### 2. `CapabilityAvailability`

Dynamic runtime availability derived from live runtime state. Each availability record must expose:
- `capabilityId`
- `availableNow`
- `degraded`
- `blockers`
- `requiredServices`
- `requiredPermissions`
- `currentBackend` or `currentMode`

### 3. `CapabilityPresentation`

External views over the same canonical source:
- `/capabilities` — capability-centric
- `/commands` — route-centric with capability references
- `/state` — lightweight runtime availability

## Capability Surface Roles

- `/capabilities` is the primary machine-readable capability-plane surface.
- `/commands` remains the route contract surface and must not carry the entire capability model alone.
- `/state` remains runtime-oriented and must expose only a lighter capability availability view.
- Shared capability meaning must come from one source of truth, not from three drifting descriptions.

## Execution Sequence

1. Land this note.
2. Move the remaining root classes into real owner packages and update manifest or imports.
3. Tighten Capability Plane v2 so the canonical model and `/capabilities` are unambiguously first-class.
4. Align `/commands`, `/state`, and `/capabilities` over the same capability source and remove stale contradictions.
5. Expand tests for root convergence and capability-surface behavior, then run the final compile and unit-test gate.
